民航订座系统升级建议

依据《民航客票销售订座系统_需求文档》和现有 hangkong.zip 分析，判断是继续改旧包，还是重新做一个轻量版桌面应用。

解压约2.2 GB
现有形态Web 快照
推荐方向轻量重写

先给结论

如果只是想让它在 Windows 桌面上双击打开，套一个 Electron 壳最快。可是要达到需求文档里的目标，继续改旧包并不省事。更稳的做法是：重新做一个轻量版 Web 系统，再打包成 Windows 桌面应用；旧包只作为参考和数据来源。

旧包适合做什么

演示基础航班、用户、订单和余额支付流程。

旧包不适合做什么

直接扩成完整民航订座系统。缺旅客类型、舱位、规则、电子客票和桌面化。

推荐做法

按需求文档重做 P0/P1/P2，最后用桌面壳打包。

需求文档要求什么

需求文档定位是教学/实训用的民航客票销售模拟系统。重点不是新闻、留言、收藏，而是“查航班、选舱位、填乘机人、确认订单、看规则、生成电子客票”。

首页查询

城市搜索、日期、成人/儿童/婴儿人数，出发到达不能相同。

航班列表

航班号、机场、时间、时长、机型、余票、舱位、价格、航司 logo，可排序。

客票预订

成人、儿童、婴儿、无成人陪伴儿童不同表单和校验。

规则展示

行李、退票、改签、特殊旅客规则，预订时能看到。

订单管理

订单列表、详情、取消、改签、退票金额模拟计算。

电子客票

PNR、票号、二维码区域、行程单打印和 PDF 保存。

响应式界面

PC、平板、手机都能正常用。

交付物

源码、数据库设计、学生说明书、教师/管理员手册、测试账号。

现有系统和需求的差距

模块现状说明
首页查询部分满足现有前台有首页和航班列表，但查询条件只有始发/目的地等基础信息；缺旅客类型人数、日期联动、城市三字码。
城市数据不满足需求要求不少于50个城市、城市代码、省份、机场名称；现有主要是航班表里的始发/目的地字符串。
航班列表部分满足现有航班列表有航班号、起飞时间、始发站、目的地、票价、图片；缺到达时间、飞行时长、航司logo、余票、舱位多价格。
成人票预订部分满足能生成订单和支付，但没有乘机人信息页、证件有效期、联系人、运输规则摘要。
儿童/婴儿票不满足没有旅客类型和年龄规则。
无成人陪伴儿童UM不满足没有UM标识、送机人、接机人、特殊说明。
订单管理部分满足有订单列表、支付、退款；缺订单号、电子客票、打印、改签、按规则计算退票金额。
客票运输规则不满足缺行李、退票、改签、特殊旅客规则展示与校验。
电子客票/行程单不满足没有PNR、票号、二维码区域、PDF行程单模板。
响应式和美化部分满足有前台页面，但风格偏模板化，移动端体验和民航行业视觉需要重做。
Windows桌面打开不满足现在是bat启动本地Web环境，不是普通用户双击打开的桌面应用。
安全和交付不满足存在默认弱口令、旧依赖、完整运行环境混装、个人信息样例等问题。

三种方案怎么选

最快

方案 A：给旧系统套一个 Windows 壳

用 Electron 做一个桌面入口，启动本地 Tomcat 和 MySQL，再自动打开原来的网页。

  • 优点：最快能看到 .exe，改动少。
  • 缺点：需求文档里的乘机人、舱位、规则、电子客票基本还是没有。
  • 适合：只想把现有演示系统变成双击打开。
判断：只能解决“桌面打开”，不能解决“达到需求”。
不推荐

方案 B：在旧 JSP 包上继续硬改

保留 Spring MVC + JSP + MySQL，把缺的功能一项项补进去。

  • 优点：能复用一部分航班、用户、订单表。
  • 缺点：旧依赖、弱口令、动态 SQL、页面模板老，越改越重。
  • 适合：必须保留旧代码，且需求只做小修小补。
判断：表面省事，后面容易返工。
推荐

方案 C：重写一个轻量版，复用旧数据思路

按需求文档重新做核心流程，再用桌面壳打包。旧系统只拿来参考字段、测试数据和页面素材。

  • 优点：功能边界清楚，界面能一次做好，后续也好维护。
  • 缺点：第一步看起来比套壳多，但总体风险低。
  • 适合：目标是满足需求文档，而不只是能打开。
判断：要达成需求文档，重写更简单。

推荐架构：Web 为核心，Windows 桌面只是入口

需求文档本身写的是 Web 版，还要求浏览器访问和响应式布局。所以不要把系统写死成单机桌面软件。正确做法是先做好 Web，再给 Windows 打一个桌面壳。

课堂/机房模式

部署一台服务器，学生用浏览器访问。适合多人同时练习。

单机演示模式

Windows 电脑双击 exe，本机启动服务。本机数据可备份，可重置。

建议的技术路线

前端

Vue 3 或 React + Vite。界面按民航蓝为主色，橙色用于价格、按钮和重点提示。页面重新设计，不沿用旧模板。

后端

Spring Boot 或 Node.js 都可以。若团队熟 Java，建议 Spring Boot。数据库用 SQLite/H2 做单机版，服务器版再切 MySQL/PostgreSQL。

桌面壳

Electron 最快，能把 Web 页面打成 Windows exe。Tauri 更轻，但开发门槛更高。这里建议先用 Electron。

旧包复用

复用航班、用户、订单这些字段思路和测试素材。旧的 JSP、Tomcat、MySQL data 目录不建议继续作为主工程。

分阶段实施计划

第 0 阶段

定范围

确认先做 P0：首页查询、航班列表、成人票预订、订单确认。不要一开始就把 UM、改签、PDF 全塞进去。

输出范围清单和页面原型。
第 1 阶段

打底座

新建 Web 应用和本地数据结构。城市、机场、航班、舱位、旅客、订单、规则这些表先定好。

能导入 50 个城市和一批测试航班。
第 2 阶段

跑通订票

做首页查询、航班排序、舱位选择、成人乘机人填写、联系人、订单确认。

一个成人旅客能从查询走到生成订单。
第 3 阶段

补规则和票据

加入儿童/婴儿校验、行李规则、退票规则、模拟电子客票和行程单页面。

能展示票号、PNR、规则摘要，能打印或导出。
第 4 阶段

桌面化

用 Electron 打包 Windows 版。双击 exe 后自动启动本地服务，打开主界面，退出时关闭后台进程。

普通用户不需要手动启动 Tomcat/MySQL。
第 5 阶段

美化和加固

统一民航蓝+橙色视觉，补加载、空状态、错误提示、二次确认。清理默认密码和旧依赖。

通过验收清单，交付安装包和说明书。
建议先做 P0，可验收后再做 P1/P2。这样最快能看到成品，也能避免一上来把票务规则做复杂。

美化方向

首页

做成航班搜索面板，城市搜索、日期、旅客人数一屏完成。少放无关新闻和轮播。

航班列表

像订票网站一样展示时间轴、舱位价格、余票和排序。价格、余票、起降时间要醒目。

预订流程

拆成四步：选航班、填乘机人、确认订单、生成电子客票。每一步都有返回和保存。

后台

管理员只看运营需要的数据：城市、机场、航班、舱位、规则、订单、测试用户。

移动端

搜索和订单查看要能在手机上完成。表单分组，避免一屏塞太多字段。

提示文案

不用技术错误码。比如“出发城市和到达城市不能相同”“儿童旅客必须关联成人”。

安全和交付加固

  • 账号和权限

    取消 123456、root/root；管理员、教师、学生分角色；敏感操作记录日志。

  • 数据安全

    身份证号、手机号做脱敏展示；测试数据不用真实个人信息。

  • 输入校验

    证件号、年龄、手机号、邮箱、航班日期都在前后端同时校验。

  • 支付说明

    明确是模拟支付；如果以后接真钱，必须接支付网关和回调验签。

  • 桌面运行

    应用只监听 127.0.0.1；数据放到用户目录；提供备份和恢复按钮。

  • 依赖升级

    不要继续用旧 Tomcat/JDK/Fastjson/Log4j 作为新基线。

最终建议

目标如果只是“能双击打开”，改旧包最快。目标如果是“达到需求文档”，建议重写轻量版。我的判断：做方案 C，先交 P0 桌面版，再逐步补 P1/P2。

最低可交付版本

Windows exe、首页查询、航班列表、成人票预订、订单确认、基础后台、测试数据。

完整目标版本

儿童/婴儿/UM、运输规则、取消/改签/退票、电子客票、行程单 PDF、响应式 Web。
